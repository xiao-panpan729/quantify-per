## 实验 #1 — 基础三元组抽取管道搭建 (2026-06-21)

### 背景
用户最初想用公众号文章训练小模型做知识图谱，经过多轮讨论后确认：直接用 API 抽取三元组，不训练模型。目标是先用 DeepSeek V4 Flash API 跑 12 个公众号 1594 篇文章，评估抽取质量。

### 设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 模型 | DeepSeek V4 (deepseek-chat) | NVIDIA V4 Flash 免费版持续 429，DeepSeek 官方 API 便宜且稳定 |
| 抽取格式 | 4列 `实体1\|关系\|实体2\|实体2类型` | 简短干净，模型只做一件事质量更稳 |
| 实体2类型 | 公司/产品/行业/概念/事件/政策/技术 | 用户确认添加类型标注不降低质量 |
| 关系类型 | 生产/供应/涨价/降价/扩产/上市/采购/利好/利空/管制/制裁/替代/出口/进口/需求/投资/合作 | 覆盖财经文章主要关系 |
| 并行策略 | 3 线程，0.5s 间隔 | 限速为主要瓶颈，多发无益 |
| 命名规则 | 公司全称、产品标准名、政治实体正式名 | 减少实体歧义 |

### 成本估算
- 每篇文章 ~700 tokens 输入 + ~50 tokens 输出
- 1594 篇 ≈ 1.1M 输入 + 80K 输出
- DeepSeek V4 价格: ~0.5 元/1M 输入, ~2 元/1M 输出
- 总成本: ~0.7 元

### Prompt 迭代
- v1: 基础 4 列格式，不加实体类型 → 实体名不规范（如"中方"不标准化）
- v2: 加命名规则（全称/标准名/正式名）→ 实体名质量提升
- v3: 加实体2类型列 → 用户确认保留

### API 测试
```
输入: 六氟化钨价格暴涨，利好国产厂商。
输出: 六氟化钨 | 利好 | 国产厂商
(39 tokens, 9 输出)
```

### 状态
- ⏸️ 1594 篇文章分批抽取中，已跑前 800 篇（中途暂停）
- 成功率 >99%（799/800 成功）

## 实验 #2 — WorkBuddy 因果边框架讨论 (2026-06-21)

### 背景
用户与 WorkBuddy 讨论了知识图谱的根本局限性，提出三元组太"薄"（只含实体-关系-实体），缺少交易决策所需的关键维度。

### WorkBuddy 的核心批评
1. **强度不是二元的** — "地缘冲突→黄金涨"是 +3%/周，不是 0/1
2. **时滞是因果的核心信息** — 原油涨→化工 3 天后跟涨，这 3 天是交易窗口
3. **因果关系是条件性的** — "央行加息"+"通胀>3%"→利空 vs "央行加息"+"通胀<2%"→影响微弱
4. **完整的因果链有结构** — 三元组把链拆碎成 5 个独立片段

### 改进方向：Causal Edge 格式
```json
{
  "id": "chain_oil_to_chem_001",
  "source": "原油价格上涨",
  "target": "化工板块",
  "direction": "正向传导",
  "strength": 0.7,
  "delay_days": 3,
  "half_life_days": 14,
  "conditions": [{"var": "原油涨幅", "op": ">", "value": "5%"}]
}
```

### 核心哲学
- **失败比成功更有价值** — 每次边失效都是补 conditions 的机会
- **图谱越用越聪明** — 不是因为加了新知识，而是精确标定了旧知识的有效范围

### 当前共识
先跑完基础三元组，看看实体和关系的覆盖质量，再决定是否升级到 causal edge 格式。

## 实验 #3 — V1→V3 设计转向：从静态三元组到 Dorian 六步框架 (2026-06-21)

**废止**: 实验 #1 的静态三元组抽取方向（4列"实体1|关系|实体2|实体2类型"，格式正确但9%有用率）
**修订**: 只抽取"变量变化/传导关系/交易指向"三类，每条带时间，主语在前

### 背景
实验 #1 全量跑完 1594 篇文章后，用户人工标记 100 条样本做质量评估：
- 有价值 9% — 全是涨价/降价类
- 噪声 25% — 格式对但无交易价值
- 错误 66% — 实体不对/关系反了/模型自指

根因：静态三元组"京东需求电商平台"这类套套逻辑无交易价值。模型不知道"什么对交易有用"。

### 设计转向

| 维度 | V1 (已废止) | V3 (当前) |
|------|------------|-----------|
| 抽取内容 | 实体1→关系→实体2→实体2类型 | 变量变化/传导关系/交易指向 |
| 时间 | 无 | 每条带 YYYY-MM-DD |
| 主语位置 | 实体1(随意) | 操作对象在前(主语|逻辑|类型|补充|时间) |
| 是否区分变量vs观点 | 不区分 | 变量变化≠交易指向 |
| 对标框架 | 通用知识图谱 | Dorian 六步(变量拆解/传导链/交易含义) |

### 三段式格式（定型）

```
变量变化: DRAM价格 | 涨 | 变量变化 | Q1涨60-70% | 2026-01-08
传导: 存储涨价 | → | 消费电子BOM成本 | 正向传导 | 存储占BOM 20-30% | 2026-01-08
交易指向: 存储芯片板块 | 利好 | 交易指向 | DRAM涨价增厚利润 | 2026-01-08
```

### 最终结果
- 全量 1594 篇文章处理完成，成功率 >99%
- 去重后 **6,308 条**
  - 变量变化: ~2,600 条（价格涨跌/产能增减/政策变化）
  - 传导关系: ~1,600 条（A→B因果链，带正向/负向标注）
  - 交易指向: ~2,100 条（利好/利空具体标的，带原因）
- 文件: `signals/tracking/_kg/v3_deduped.json`
- 耗时: ~60 分钟 / 14 批次 × 100 篇

### 关键设计决策
1. **主语在前**: 一眼扫过就知道"谁变了"，不需要从时间戳往后读
2. **时间在尾**: 不干扰阅读但每条都有时间锚点，过期判断可行
3. **利好利空复归**: V2 曾一刀切删除，V3 恢复作为"传导的终点"(所以然的输出)
4. **无条件语不输出**: 没有明确变化的静态关系(XX需求XX)不浪费 token

### 遗留问题
- 部分日期推断不准(文章无明确日期→模型用"近期"或截断)
- 变量名标准化问题：同一变量可能用不同名称(DRAM价格/存储价格)
- 交易指向的利好/利空是文章观点，未经交叉验证

> ⚠️ 时间在尾已废止，见实验 #4

## 实验 #4 — V3→V4 格式升级 + 质量评估 (2026-06-22)

**废止**: 实验 #3 的时间在尾设计（用户确认时间必须前置，扫一眼就知道是不是旧闻）
**修订**: 时间嵌入主语头部 + 主语必须是"变量变化"而非静态实体 + 交易指向原因/标的分离

### 背景

V3 输出 6308 条后，用户对 100 条做抽样标注，发现三类根因问题：

| 问题 | 占比 | 表现 |
|------|------|------|
| 因果倒置/主语错误 | ~40% | `宁德时代 \| 利好 \| 交易指向` — 静态实体做主语，原因跑去了补充区 |
| 纯行情报价/无量化 | ~10% | `COMEX黄金 \| 跌 \| 1.12%` — 行情软件就能看，无因无果 |
| 时间不可见 | ~100% | 时间藏在行尾，扫一眼不知道是否过期 |

### 格式规格文档化

创建 `signals/tracking/_kg/KG_FORMAT_SPEC.md`，5 章完整规格：

1. **四核心原则**:
   - 主语 = 变量变化，不是静态实体（`电池环节引领估值提升 \| 利好 \| ...` 而非 `宁德时代 \| 利好 \| ...`）
   - 时间前置（`2026-05-22 DRAM涨价 \| ... \| 变量变化`）
   - 变量变化必须量化（无具体数字不如不提取）
   - 事实锚点 ≠ 交易指向（两条分开写）

2. **三种类型的完整格式**:
   - 变量变化: `[时间] [标的+变量+数值] \| [涨/跌/增/降] \| 变量变化`
   - 交易指向: `[时间] [原因] \| [利好/利空] \| 交易指向 \| [具体标的]`
   - 传导: `[时间] [source] \| → \| [target] \| [方向] \| [详情]`

3. **排除规则**: 纯行情报价 / 无数据主观观点 / 无幅度变化 / 笼统传导终点

4. **二维标注体系**: `__标注__`（逻辑是否正确）vs `__市场验证__`（事后行情检验），两条独立互不替代

5. **V3 质量评估**: 基于 30 条抽样标注的量化分布

### 全量处理结果

使用结构模式匹配（非关键词脚本）对 6308 条全量处理：

| 操作 | 数量 |
|------|------|
| 交易指向因果调换 | 1002 条 |
| 变量变化主语补全 | 2268 条 |
| 时间前置 | 5073 条 |
| 删除（纯报价 27 / 无幅度 45 / 笼统终点 16 / 笼统传导 3） | 91 条 |
| **最终保留** | **6,217 条** |

输出文件: `signals/tracking/_kg/v4_formatted.json` (version: v4_final)

### 关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 检测方法 | 语法结构模式（正则+长度+数值检测） | 用户否决关键词脚本方案——"写脚本是不对的，我要充分利用你的智能" |
| 删除标准 | 4 类明确规则（纯报价/无幅度/笼统终点/笼统传导） | 不过滤年份（用户不同意按时间删除） |
| 格式修正 | 全部在 display 字段原地修改 | 保留原始数据，只改展示层 |
| 标注体系 | 双维度（逻辑正确 vs 市场验证） | "逻辑对但市场没认"也是有用信息 |

### 重要发现：格式≠质量

V4 处理让 6217 条在**格式上**合规了，但用户抽查 10 条后发现大量"格式对了但内容没价值"的条目。典型表现：

- **过期数据**（2021-2022 年的传导，时过境迁）
- **轱辘话**（主语和补充区一模一样或内容重叠）
- **定性描述冒充变量**（"风险增加"无量化、"成功"无定义）
- **纯 K 线报价**（指数涨跌无因果）

这些靠规则脚本检测不了，需要用户的人工标注判断。

## 实验 #5 — SAG 方向转向：从三元组到动态 SQL 实体-事件索引 (2026-06-25)

**废止**: 实验 #2/#3/#4 的 Causal Edge 方案（用户阅读 SAG 论文后决定彻底转向）
**修订**: 不建离线知识图谱，用 SQL 数据库存事件+实体，查询时动态 JOIN 构建超边

### 背景

用户阅读了 **SAG (SQL-Retrieval Augmented Generation)** 文章（Zleap-AI/SAG on GitHub），核心洞察：

> 传统知识图谱（三元组/因果边）是**预建静态结构**，而投资分析需要的是**查询时动态关联**。

SAG 的核心思想：
1. 不预先构建知识图谱，而是将实体和事件存入 SQL 数据库
2. 在查询时通过 SQL 的 `JOIN` 操作动态发现实体间关系
3. 使用动态超边（Dynamic Hyperedges）替代预定义的三元组
4. 大模型在查询时通过 SQL 结果构建上下文，而非从静态图谱检索

### 与当前系统的关系

```
当前 (kg_extract.py)              未来 (SAG 方案)
─────────────────                ──────────
公众号文章 → 三元组抽取            公众号文章 → 事件 + 实体 → SQL 存储
三元组 → 知识图谱文件              查询时 SQL JOIN → 动态关系发现
静态图谱 → 检索 → LLM             SQL 结果 → LLM 推理
```

### 对现有 kg_extract.py 的影响

| 方面 | 原来 | 转向后 |
|------|------|--------|
| 输出格式 | 三元组 (实体1\|关系\|实体2) | 事件记录 + 实体索引 (可 SQL 查询的结构化行) |
| 存储 | JSON 文件 | SQLite/数据库 |
| 查询 | 文件遍历/关键词匹配 | SQL JOIN + 动态超边 |
| 更新 | 全量重新抽取 | 增量 INSERT |
| 与 LLM 接口 | 检索已有图谱 | SQL 结果作为上下文 |

### 数据源整合

SAG 数据库将统一两个管道的数据：
- **知识星球**（47,245 篇话题，来自 kb_capture_log.md 实验 #1）
- **公众号文章**（6,217 条 V4 三元组，来自实验 #4）

两者都入库后，查询时可通过 SQL JOIN 跨源发现关联。
